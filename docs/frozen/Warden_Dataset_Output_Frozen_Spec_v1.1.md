# Warden Dataset Output Frozen Spec v1.1

This document freezes the current on-disk sample-output structure produced by the capture and labeling scripts.

It is the dataset-structure baseline for later data processing, backfill, statistics, training, and review scripts.

## 1. Purpose

This file freezes the current script-produced sample layout, including:

- which files a successful sample directory may contain;
- what the main `json / txt / png / html` artifacts mean;
- the active structure of `auto_labels.json` and `rule_labels.json`;
- which files are generated by default and which are optional;
- which fields should stop changing casually unless the spec version is explicitly bumped.

In practical terms, later dataset scripts, backfill scripts, statistics scripts, and training scripts should treat this file as the upstream structure contract.

## 2. Freeze Principles

### 2.1 Current Script Reality Wins

If older notes, earlier drafts, or previous chats disagree with the current script outputs, the current script outputs win.

### 2.2 Scope Of The Freeze

The frozen scope includes:

- sample directory structure;
- file names;
- JSON top-level field names;
- `auto_labels.json` field families;
- `rule_labels.json` field families;
- optional `after_action/` and `variants/` layout.

### 2.3 When Changes Are Allowed

Only two cases should change this contract:

1. reviewer or publication requirements make a change unavoidable;
2. the specification is explicitly version-bumped, such as `v1.2` or `v2.0`.

Otherwise:

- do not silently rename fields;
- do not silently change enumerations;
- do not remove old fields without documentation.

## 3. Top-Level Sample Directory Structure

A successful sample directory may contain core files such as:

- `meta.json`
- `url.json`
- `env.json`
- `redirect_chain.json`
- `html_raw.json`
- `html_rendered.json`
- `visible_text.txt`
- `forms.json`
- `screenshot_viewport.png`
- `screenshot_full.png`
- `net_summary.json`
- `auto_labels.json`
- `rule_labels.json`
- `manual_labels.json`

Mode-dependent or optional artifacts include:

- `actions.jsonl`
- `after_action/`
- `variants/`
- `diff_summary.json`
- `network.har`

The main point is that the sample directory layout is frozen even when not every optional artifact exists in every mode.

## 4. Generation Conditions

### 4.1 Only Successful Samples Create Sample Directories

Under the current capture pipeline, failed or clearly invalid samples do not become full successful sample directories.

### 4.2 Defaults Versus Optional Modes

Under the current implementation:

- `html_raw.json`, `html_rendered.json`, `visible_text.txt`, `forms.json`, `auto_labels.json`, and `net_summary.json` are normally enabled;
- `screenshot_full.png` is normally attempted but may be absent if the page is too large or the full-page capture fails;
- `actions.jsonl`, `after_action/`, `variants/`, `diff_summary.json`, and `rule_labels.json` are mode-dependent and not always emitted by default.

## 5. Sample Directory Naming And `sample_id`

The sample directory name is built from a sanitized URL-derived folder name plus a UTC timestamp.

`meta.json.sample_id` is expected to match the sample directory name directly.

## 6. File-Level Structure Summary

This section explains the purpose and active meaning of the main files. Exact examples and field lists remain frozen in the Chinese source below.

### 6.1 `meta.json`

`meta.json` stores the basic sample metadata.

Important semantics:

- `sample_id` is the directory identifier;
- `label` is only a source hint such as `phish` or `benign`, not a human gold label;
- `crawl_time_utc`, `http_status`, and related metadata describe the capture event rather than a review decision.

### 6.2 `url.json`

`url.json` stores the normalized input URL, the final URL, and the redirect-chain view of the request path.

### 6.3 `env.json`

`env.json` stores capture-environment parameters such as browser channel, headless mode, viewport, JavaScript setting, service-worker policy, proxy setting, and user agent.

### 6.4 `redirect_chain.json`

`redirect_chain.json` stores the redirect chain as a standalone array so downstream consumers do not have to read it only through `url.json`.

### 6.5 `html_raw.json`

`html_raw.json` stores a JSON wrapper for the raw main-document response body.
The HTML text is stored inside the wrapper payload and may be truncated according to the active HTML-length limit.

### 6.6 `html_rendered.json`

`html_rendered.json` stores a JSON wrapper for the browser-rendered DOM content.
It may differ from `html_raw.json` and remains one of the main downstream text-analysis sources.

### 6.7 `visible_text.txt`

`visible_text.txt` stores the extracted visible text. It is one of the main inputs for text towers and offline label backfill.

### 6.8 `forms.json`

`forms.json` stores page forms and their input structure, including action URLs, methods, individual input attributes, and summary flags such as password, OTP, or card-related evidence.

### 6.9 `screenshot_viewport.png`

`screenshot_viewport.png` is the viewport screenshot and is expected for successful samples. It is the main frozen visual artifact for review, OCR-style processing, and lightweight visual modeling.

### 6.10 `screenshot_full.png`

`screenshot_full.png` is the full-page screenshot. It is optional and its absence does not automatically mean the sample failed.

### 6.11 `net_summary.json`

`net_summary.json` is the lightweight network-evidence summary.

One important rule is frozen here:

- the file name is stable;
- the internal structure depends on the active network-evidence mode.

Under the default lite mode it summarizes request counts, domains, POST targets, timing, navigation, anomalies, and collection errors.

Under HAR-derived mode it becomes a summary generated from `network.har`.

### 6.12 `network.har`

`network.har` is optional raw HAR evidence. It is not the default output in the lite-mode path, but if HAR mode is enabled it may become the upstream evidence source for `net_summary.json`.

### 6.13 `actions.jsonl`

`actions.jsonl` records lightweight step-one interaction candidates and the final decision of that step when the optional step-one interaction mode is enabled.

### 6.14 `after_action/`

`after_action/` stores page snapshots after a successful lightweight interaction step, such as post-click screenshots, rendered HTML, visible text, forms, and network summary.

### 6.15 `variants/` And `diff_summary.json`

`variants/` stores alternate environment captures. `diff_summary.json` summarizes important differences across the base sample and variants, such as text changes, form changes, network differences, redirect differences, and possible cloaking flags.

### 6.16 `auto_labels.json`

`auto_labels.json` is the main automatically generated weak-label layer.

The frozen top-level structure includes fields such as:

- `schema_version`
- `generated_at_utc`
- `source`
- `label_hint`
- `page_stage_candidate`
- `language_candidate`
- `url_features`
- `form_features`
- `html_features`
- `brand_signals`
- `intent_signals`
- `evasion_signals`
- `network_features`
- `risk_outputs`

Important frozen semantics include:

- current implementation reality wins over older schema drafts;
- `page_stage_candidate` must follow the active implemented enumeration;
- `language_candidate` is a coarse script or language-family guess, not a strict language-ID result;
- `url_features` currently uses `is_ip_host`, not an older draft such as `has_ip_host`;
- `html_features` currently emphasizes scripts, libraries, obfuscation, download evidence, and CAPTCHA-like evidence rather than a maximal HTML semantic inventory.

### 6.17 `rule_labels.json`

`rule_labels.json` is the rule-derived layer built on top of `auto_labels.json`.

Important semantics:

- it may be emitted offline by backfill or optionally during capture;
- it can store `review_priority`;
- it can store `threat_taxonomy_v1` as a long-lived weak-label namespace;
- `primary_threat_label_candidate` and related confidence or rule fields remain weak rule-derived outputs, not human gold labels.

### 6.18 `manual_labels.json`

`manual_labels.json` is the recommended human-label layer.

It is not automatically produced by the current capture or backfill scripts. It is a recommended structure for later human annotation.

## 7. Offline Backfill Script Inputs And Outputs

The offline backfill path reads existing frozen capture artifacts and emits label files without rewriting the core capture files.

The backfill script reads files such as:

- `meta.json`
- `url.json`
- `forms.json`
- `net_summary.json`
- `diff_summary.json`
- `visible_text.txt`
- `html_rendered.json`
- or falls back to `html_raw.json` when needed.

It emits:

- `auto_labels.json` by default;
- `rule_labels.json` when rule-label emission is enabled.

It does not rewrite:

- `meta.json`
- `url.json`
- `env.json`
- `redirect_chain.json`
- screenshot files
- `manual_labels.json`

## 8. Brand-Lexicon Loading Rules

The brand-lexicon loading rule is also part of the active contract for the current label pipeline.

The priority order is:

1. an explicitly provided lexicon path;
2. auto-discovery of nearby lexicon files such as `evt_brand_lexicon_v1.json` or `evt_brand_lexicon.json`;
3. a built-in small fallback lexicon.

Brand evidence currently comes from:

- `text_brand_candidates` derived from title and visible text;
- `url_brand_candidates` derived from host tokens and restricted typo matching;
- `claimed_brands` as the merged and deduplicated result.

## 9. Differences From Older Schema Drafts

One rule should be treated as non-negotiable:

current implementation is the data truth.

In particular:

- `page_stage_candidate` must follow the current implemented enumeration;
- `url_features` currently uses the field name `is_ip_host`;
- `html_features` currently reflects script, library, obfuscation, download, and CAPTCHA-oriented evidence more than a broad semantic inventory;
- `manual_labels.json` is a recommended structure rather than an automatically produced artifact.

## 10. Freeze Recommendation

From this version onward, this document should be treated as the frozen specification for:

- real on-disk capture output structure;
- real auto-label output structure;
- real rule-label output structure;
- recommended manual-label structure.

If the project needs to change this contract later, the correct order is:

1. update the spec version;
2. update the scripts;
3. update backfill and training tooling.

The project should not silently change scripts first and let documentation chase behind later.

## 11. Final Conclusion

This file freezes:

- the real current sample file structure;
- the real current automatic weak-label structure;
- the real current rule-derived output structure;
- the recommended manual-label structure.

From this version onward, the default assumption is that the dataset structure should remain stable unless a reviewer requirement or an explicit version bump forces a change.

## 6. 文件级结构说明

---

## 6.1 `meta.json`

用途：记录样本基础元信息。

### 实际字段

```json
{
  "sample_id": "<folder_name>",
  "label": "phish | benign",
  "crawl_time_utc": "YYYYMMDDTHHMMSSZ",
  "http_status": 200,
  "page_title": "...",
  "etld1_mode": "..."
}
```

### 字段说明

- `sample_id`：样本目录名
- `label`：根目录标签提示，一般来自 crawler 当前处理的集合（如 `phish/benign`），**不是人工金标**
- `crawl_time_utc`：抓取时间（UTC，紧凑格式）
- `http_status`：主文档响应状态码
- `page_title`：页面标题
- `etld1_mode`：eTLD+1 提取模式（脚本内部配置）

---

## 6.2 `url.json`

用途：记录输入 URL、最终 URL 与原始跳转链。

### 实际字段

```json
{
  "input_url": "https://...",
  "final_url": "https://...",
  "redirect_chain": [
    "https://...",
    "https://..."
  ]
}
```

### 字段说明

- `input_url`：原始输入 URL（归一化后）
- `final_url`：页面最终落点 URL
- `redirect_chain`：按请求跳转顺序记录的 URL 字符串数组

---

## 6.3 `env.json`

用途：记录抓取环境参数。

### 实际字段

```json
{
  "browser_channel": "chrome",
  "headless": true,
  "viewport": {"width": 1365, "height": 768},
  "java_script_enabled": true,
  "service_workers": "block",
  "ignore_https_errors": false,
  "bypass_csp": false,
  "proxy_server": null,
  "user_agent": "..."
}
```

### 字段说明

- `browser_channel`：浏览器通道
- `headless`：是否无头
- `viewport`：窗口大小
- `java_script_enabled`：是否启用 JS
- `service_workers`：service worker 策略
- `ignore_https_errors`：是否忽略 HTTPS 错误
- `bypass_csp`：是否绕过 CSP
- `proxy_server`：代理地址；无代理时通常为 `null`
- `user_agent`：页面实际 UA

---

## 6.4 `redirect_chain.json`

用途：单独保存跳转链，方便下游直接读取。

### 实际结构

```json
[
  "https://first-hop.example/...",
  "https://second-hop.example/...",
  "https://final.example/..."
]
```

说明：它与 `url.json.redirect_chain` 含义一致，只是单独落一份文件，方便训练/统计直接消费。

---

## 6.5 `html_raw.json`

用途：主文档原始响应体（raw HTML）的 JSON 封装。

### 特点

- 来自响应体 `resp.text()`
- HTML 文本保存在 JSON payload 内
- 可能包含 `<!-- TRUNCATED -->` 标记
- 长度受 `MAX_HTML_CHARS` 限制

---

## 6.6 `html_rendered.json`

用途：浏览器渲染后 DOM 内容的 JSON 封装。

### 特点

- 来自 `page.content()`
- HTML 文本保存在 JSON payload 内
- 可能与 `html_raw.json` 不同
- 长度受 `MAX_HTML_CHARS` 限制

---

## 6.7 `visible_text.txt`

用途：页面可见文本抽取结果。

### 特点

- 默认包含标题 + 页面主体可见文本
- 若正文很空，会回退收集 `h1/h2/h3/button/a/label/p/span`
- 长度受 `MAX_VISIBLE_TEXT_CHARS` 限制

这份文件是当前文本塔 / 离线补标的重要输入之一。

---

## 6.8 `forms.json`

用途：页面表单与输入字段结构。

### 实际结构

```json
{
  "forms": [
    {
      "action": "...",
      "action_abs": "https://...",
      "method": "GET | POST | ...",
      "inputs": [
        {
          "tag": "input | textarea | select",
          "type": "text | password | email | ...",
          "name": "...",
          "id": "...",
          "autocomplete": "...",
          "placeholder": "...",
          "aria_label": "..."
        }
      ],
      "has_password": true,
      "has_otp": false,
      "has_card": false
    }
  ]
}
```

### 字段说明

#### 顶层
- `forms`：页面中所有 `form` 的数组

#### 单个 form
- `action`：原始 action
- `action_abs`：绝对化后的 action URL
- `method`：表单提交方法
- `inputs`：表单内部所有 `input/textarea/select`
- `has_password`：表单内是否存在密码相关证据
- `has_otp`：表单内是否存在 OTP 相关证据
- `has_card`：表单内是否存在卡支付相关证据

#### 单个 input
- `tag`
- `type`
- `name`
- `id`
- `autocomplete`
- `placeholder`
- `aria_label`

用途：首屏截图。

说明：

- 成功样本必有
- 通过了基本截图质量闸门
- 是视觉后续人工复核、OCR、轻量视觉摘要的原始依据

## 6.10 `screenshot_full.png`

用途：整页截图。

说明：

- 默认尝试保存
- 若页面过高或 full-page 截图失败，可能不存在
- 不存在不代表样本失败

## 6.11 `net_summary.json`

用途：轻量网络证据摘要。

当前脚本有三种来源：

1. 默认：`NetEvidenceLiteCollector` 在线统计得到摘要  
2. 若改成 HAR 模式：由 `network.har` 再汇总  
3. 若网络证据关闭：写入 `{"note": "net_summary_disabled"}`

### 默认 lite 模式下的实际结构

```json
{
  "final_url": "https://...",
  "top_frame_final_origin": "https://...",
  "request_counts": {
    "total": 123,
    "by_resource_type": {"document": 1, "script": 12},
    "by_method": {"GET": 120, "POST": 3}
  },
  "domain_counts": [
    {
      "domain_etld1": "example.com",
      "count": 10,
      "methods": {"GET": 9, "POST": 1}
    }
  ],
  "third_party_domains": ["cdn.example.net", "evil.com"],
  "post_targets": [
    {"domain_etld1": "evil.com", "count": 2}
  ],
  "status_counts": {...},
  "timing_ms": {
    "ttfb_p50": 100.0,
    "ttfb_p95": 300.0,
    "response_end_p50": 500.0,
    "response_end_p95": 1500.0
  },
  "navigation_chain": [
    {"url": "https://...", "status": 301},
    {"url": "https://...", "status": 200}
  ],
  "anomalies": ["many_third_party", "post_to_third_party"],
  "errors": ["..."]
}
```

### 字段说明

- `final_url`：最终 URL（安全摘要形式）
- `top_frame_final_origin`：最终 origin
- `request_counts.total`：请求总数
- `request_counts.by_resource_type`：按资源类型计数
- `request_counts.by_method`：按 HTTP 方法计数
- `domain_counts`：按 eTLD+1 聚合的请求计数和方法统计
- `third_party_domains`：与顶层站点不同的第三方域列表
- `post_targets`：POST 目标域及计数
- `status_counts`：HTTP 状态码分组统计
- `timing_ms`：网络时延分位数

## 6.17 `rule_labels.json`

用途：基于 `auto_labels.json` 的规则补齐结果。

### 生成方式

- 离线补标脚本：`--emit-rule-labels`
- 在线采集脚本：只有 `SAVE_RULE_LABELS=True` 才生成

### 结构

```json
{
  "schema_version": "evt_v1.1",
  "generated_at_utc": "2026-03-16T...Z",
  "rule_flags": {
    "escalate_to_l2_candidate": true,
    "brand_mismatch_with_sensitive_intent": true,
    "off_domain_sensitive_form": true
  },
  "review_priority": "p0 | p1 | p2 | p3",
  "threat_taxonomy_v1": {
    "primary_threat_label_candidate": "credential_theft | payment_fraud | wallet_drain_or_web3_approval_fraud | pii_kyc_harvesting | fake_support_or_contact_diversion | malware_or_fake_download | benign | uncertain",
    "primary_threat_label_confidence": 0.0,
    "primary_threat_label_rules": ["..."],
    "scenario_label_candidate": "finance_banking | ecommerce_retail | payment_platform | logistics_delivery | enterprise_mail_cloud | social_media | government_public_service | crypto_web3 | gaming | telecom_utility | tech_support | job_recruitment | other",
    "scenario_label_confidence": 0.0,
    "scenario_label_rules": ["..."],
    "narrative_tags_candidate": ["brand_impersonation"],
    "evidence_tags_candidate": ["credential_form_present"],
    "evasion_tags_candidate": ["requires_interaction_to_reveal"],
    "ecosystem_tags_candidate": ["illicit_service_content"],
    "taxonomy_source": "rule_derived_from_auto_labels",
    "taxonomy_review_status": "weak_candidate_only"
  }
}
```

### 字段说明

- `escalate_to_l2_candidate`：是否应进入高层复核
- `brand_mismatch_with_sensitive_intent`：品牌不一致 + 敏感意图
- `off_domain_sensitive_form`：异域提交 + 敏感表单
- `review_priority`：
  - `p0`：`risk_level_weak == critical`
  - `p1`：`high`
  - `p2`：`medium`
  - `p3`：其余
- `threat_taxonomy_v1`：长期保留在 `rule_labels.json` 下的活跃弱标签命名空间，不是临时实验字段
- `primary_threat_label_candidate`：一级主威胁候选标签，不等同于人工金标
- `primary_threat_label_confidence`：规则派生置信度，属于启发式分数，不是校准概率
- `primary_threat_label_rules`：触发该候选标签的规则来源列表
- `scenario_label_candidate`：行业 / 场景外壳候选标签
- `scenario_label_confidence`：场景候选的规则派生置信度
- `scenario_label_rules`：触发该场景候选的规则来源列表
- `narrative_tags_candidate`：社工叙事候选标签列表
- `evidence_tags_candidate`：显式证据候选标签列表
- `evasion_tags_candidate`：对抗 / 规避候选标签列表
- `ecosystem_tags_candidate`：生态 / 内容属性候选标签列表
- `taxonomy_source`：固定标记为规则派生来源
- `taxonomy_review_status`：固定弱标签安全语义，当前默认值为 `weak_candidate_only`

### 使用边界

- `threat_taxonomy_v1` 应继续通过统一离线 backfill 提高覆盖率
- 它承载 Warden 多威胁主问题定义，但不直接提升为 TrainSet V1 primary 默认金标
- 当前 primary manifest 核心字段不默认展开写入该命名空间
