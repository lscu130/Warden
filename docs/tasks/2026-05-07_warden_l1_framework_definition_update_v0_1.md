- L1 的总体定位；
- 文本塔职责；
- 多任务头设计方向；
- 视觉路径职责；
- CLIP / YOLO / OCR 的分工；
- Fusion / XGBoost 的职责；
- 证据账本与解释器的职责；
- L1 输出应包含的最终判断、风险轴、页面角色、路由和解释证据。

### 当前结论

当前设计已经明确：

1. **L1 是主判断层，但不是单模型一锤定音。**
2. **L1 的主证据路径是文本 / HTML / URL / forms / network / structured signals。**
3. **文本塔不是完整推理器；文本塔应学习结构化语义概念识别和关系判断，形成“隐式 CoT”式的中间概念输出。**
4. **文本塔应采用多任务头，降低单一最终标签任务过重的问题。**
5. **视觉路径不是独立威胁判断器；视觉路径是证据恢复和局部证据定位路径。**
6. **OCR 和 YOLO 是视觉端的核心证据恢复工具：OCR 恢复截图文字，YOLO 定位动作组件 / 高危 UI 组件。**
7. **CLIP 降级为页面级视觉场景先验 / 路由辅助 / 弱证据补充，不负责最终 malicious / benign 判断。**
8. **XGBoost 如果作为 fusion head，应负责融合概念输出和结构化特征，形成 L1 最终机器判断；它不直接吃 raw text 或 raw screenshot。**
9. **L1 的解释来自 evidence ledger + reason codes + deterministic explanation renderer，不来自 BERT / CLIP / XGBoost 的自由生成。**

## 1. Background

Warden has already frozen a staged runtime/dataflow framing around `L0 / L1 / L2`.

The current design direction for L1 has been refined:

- L1 remains the main judgment stage.
- L1 should consume a source-aware evidence pack derived from `SampleContext`, not raw artifacts directly.
- Text / HTML / URL / forms / network / structured joint signals remain the main evidence path.
- The text tower is not expected to perform complete human-like reasoning. It should learn structured semantic concepts and relation judgments through multi-task heads.
- The vision path should be redefined as evidence recovery and evidence localization, not as an independent visual threat classifier.
- In the vision path, OCR and YOLO are core evidence-recovery tools. CLIP is retained only as a lightweight page-level visual prior and routing/helper signal.
- Fusion should combine text concepts, vision recovered evidence, structured features, and joint signals into the final L1 machine judgment.
- Explanation should be generated deterministically from an evidence ledger and reason codes, not from free-form text generation by BERT, CLIP, or XGBoost.

This task exists to align repository documentation with that refined L1 definition.

## 2. Goal

Update the project documentation so Warden has a consistent, concise, and implementation-ready definition of the current L1 framework.

The updated documentation must make clear that:

1. L1 is a staged main-judgment shell, not a monolithic model.
2. The text tower learns structured semantic concept recognition and relation judgment, not complete free-form reasoning.
3. The text tower uses multi-task heads to decompose action surfaces, behavior contexts, relation/consistency judgments, risk axes, page roles, and routing needs.
4. The vision path is an evidence recovery path for text/HTML/action-evidence insufficiency.
5. OCR and YOLO are the main visual evidence recovery tools.
6. CLIP is a weak page-level visual prior and routing helper, not a final threat judge.
7. Fusion, including possible XGBoost heads, produces final L1 machine outputs from concept outputs and structured/joint features.
8. L1 explanations are rendered from evidence ledger entries and reason codes, not generated by the student models.

### Prior design context to preserve

Use the following refined design as task input:

#### L1 overall

L1 is the main judgment stage, but it must not be represented as a single monolithic model.

Preferred conceptual structure:

```text
L1 main judgment shell
  ├─ source-aware evidence pack builder
  ├─ text / HTML main path
  ├─ structured feature and joint-signal branch
  ├─ visual evidence recovery path, triggered only when needed
  ├─ fusion head(s)
  └─ explanation renderer
```

#### Text tower

The text tower should not be expected to learn complete human-like reasoning.

It should learn:

- structured semantic concept recognition;
- action surface recognition;
- behavior context recognition;
- relation and consistency judgments;
- risk axis classification;
- page role classification;
- routing support.

This is the intended “implicit CoT” pattern: the student does not output a chain-of-thought, but it outputs intermediate concepts that fusion can combine.

Draft text multi-task head groups:

```text
Action Surface Heads:
  - login_surface
  - signup_surface
  - payment_surface
  - wallet_connect_surface
  - download_surface
  - support_contact_surface
  - pii_collection_surface
  - third_party_redirect_surface

Behavior Context Heads:
  - brand_impersonation_context
  - authority_impersonation_context
  - institution_impersonation_context
  - fake_security_context
  - fake_financial_context
  - fake_reward_or_prize_context
  - fake_support_context
  - urgency_or_threat_context
  - deceptive_hosted_brand_shell

Relation / Consistency Heads:
  - brand_domain_alignment
  - brand_url_token_consistency
  - claimed_brand_officiality
  - action_target_alignment
  - form_action_alignment
  - download_target_alignment
  - redirect_chain_reasonableness
  - hosted_platform_context

Risk Axis Heads:
  - human_exposure_risk
  - deceptive_identity_risk
  - observed_action_risk
  - payload_deployment_risk
  - brand_domain_conflict_risk
  - evidence_incompleteness_risk
  - evasion_or_gate_risk

Page Role Heads:
  - benign_clear
  - benign_hard_negative
  - brand_impersonation_landing_shell_without_payload_observed
```

#### Visual evidence recovery path

The visual path should be defined as evidence recovery and evidence localization, not as a standalone threat classifier.

It is triggered when text / HTML / action evidence is insufficient, such as:

- `visible_text.txt` is missing or empty;
- effective visible text is sparse, with V0.1 default around `< 300` effective characters;
- HTML actionable structure is sparse;
- forms summary is missing but screenshot appears interactive;
- SPA / React / Vue extraction is weak;
- L1 text confidence is low;
- text, URL, HTML, forms, or network evidence conflicts;
- L0 marks possible gate / evasion / brand-surface conflict.

Visual component responsibilities:

```text
OCR:
  Recovers screenshot text when visible_text and HTML text are missing, sparse, or incomplete.

YOLO:
  Detects and localizes atomic UI evidence such as input boxes, password fields, OTP/code fields,
  card/payment fields, wallet buttons, download buttons, QR codes, captcha/verify widgets,
  popups/modals, and primary CTA buttons.

CLIP:
  Provides page-level visual scene prior, visual routing assistance, and weak scene evidence.
  It may identify whether a page visually resembles login, wallet, payment, download, gate,
  brand landing, support, template/demo, or ordinary content.
  It must not produce the final malicious/benign judgment.
```

#### Fusion

Fusion should combine:

- text tower concept logits;
- relation/risk/page-role logits;
- structured features;
- joint signals;
- visual recovered evidence from OCR/YOLO;
- optional weak CLIP visual prior.

If XGBoost is used, it should not consume raw text or raw screenshot directly. It should consume concept outputs, structured features, and joint signals.

Final L1 machine outputs should include at least the following draft concepts:

```text
final_label:
  - benign
  - malicious
  - suspicious
  - unknown

malicious_basis:
  - no_malicious_evidence_observed
  - high_risk_behavior_observed
  - high_risk_action_observed
  - both_behavior_and_action_observed
  - insufficient_evidence

other outputs:
  - risk_score
  - confidence
  - payload_observed
  - page_role
  - risk_axes
  - routing decision
```

#### Explanation

L1 explanation must not be free-form generation by BERT, CLIP, or XGBoost.

Preferred explanation architecture:

```text
Evidence Pack Builder
  -> evidence ledger

Text / Vision / Structured branches
  -> concept outputs and reason-code candidates

Fusion
  -> final label, risk axes, page role, routing, top contributors

Explanation Renderer
  -> deterministic, evidence-grounded explanation
```

Evidence and explanation should be anchored by:

- source;
- claim;
- value;
- confidence when available;
- evidence ID;
- reason code;
- whether evidence supports, limits, or conflicts with the final decision.

### Failure modes to consider

- Accidentally making CLIP appear as a standalone malicious/benign classifier.
- Accidentally making OCR always-on.
- Accidentally making YOLO mandatory for all L1 samples.
- Accidentally changing machine-readable output schema in documentation without implementation.
- Confusing action surface with threat action.
- Reintroducing `brand-domain mismatch = malicious` as a one-factor rule.
- Overwriting runtime/dataflow contracts that only froze runtime boundaries, not final model logic.

### Counterexamples to preserve

- Normal official login pages are benign hard negatives, not malicious by login surface alone.
- Normal download pages and third-party platform download links are not malicious by default.
- Normal Web3 / exchange / wallet official pages may have unusual domains and should not be automatically treated as malicious.
- Brand-only or visual-similarity-only signals can create false positives.
- Pure image pages may require OCR/YOLO evidence recovery rather than CLIP threat classification.

### Alternatives to compare

- Keep CLIP as visual tower core.
- Make OCR/YOLO the core visual path and CLIP optional.
- Use text-dominant path with no CLIP in V1.
- Keep current edge defaults while redefining CLIP as weak visual prior.

### Decision rule

Accept the proposed framing only if documentation can preserve the existing staged runtime and edge deployment constraints while clarifying that CLIP is not a final threat classifier.

## 14. Suggested Target Wording For The New L1 Definition

Codex may use or adapt the following wording in the updated documentation.

### L1 architecture summary

```text
L1 is Warden's main judgment stage. It is not a monolithic classifier.
L1 consumes a source-aware evidence bundle derived from SampleContext, combines
text / HTML / URL / form / network / structured evidence, and requests heavier
visual evidence only when the active path needs it.
```

### Text tower definition

```text
The L1 text tower is not expected to perform complete free-form reasoning.
It should learn structured semantic concepts and relation judgments through
multi-task heads. These heads decompose the judgment into action surfaces,
behavior contexts, relation/consistency checks, risk axes, page roles, and
routing needs. The intended effect is an implicit reasoning scaffold: the model
does not output chain-of-thought, but it outputs intermediate concepts that
fusion can combine into a final L1 decision.
```

### Visual evidence recovery definition

```text
The L1 visual path is an evidence recovery and evidence localization path, not
an independent visual threat-judgment path. It is triggered when visible text,
HTML/actionable structure, or other cheap evidence is sparse, missing,
low-confidence, or conflicting.
```

### OCR / YOLO / CLIP responsibility split

```text
OCR recovers screenshot text when visible_text and HTML text are missing,
sparse, image-rendered, or incomplete.

YOLO detects and localizes atomic UI evidence such as input boxes, password or
OTP fields, payment fields, wallet/connect buttons, QR codes, download buttons,
captcha/verify widgets, popups/modals, and primary CTA buttons.

CLIP is retained as a lightweight visual-prior module. It provides page-level
visual scenario scoring and auxiliary routing signals for detector/OCR
execution. CLIP must not be treated as a standalone malicious/benign classifier
and must not produce the final L1 threat verdict.
```

### Fusion definition

```text
Fusion combines text concepts, relation logits, risk-axis logits, structured
features, joint signals, OCR-recovered text evidence, YOLO-localized UI
evidence, and optional weak CLIP visual priors. If XGBoost is used as a fusion
head, it should consume concept outputs and structured/joint features rather
than raw webpage text or raw screenshot pixels.
```

### Explanation definition

```text
L1 explanations are produced from an evidence ledger, reason codes, and a
deterministic explanation renderer. Student models such as BERT, CLIP, and
XGBoost do not generate free-form explanations. They activate concepts and
scores; the renderer converts evidence-grounded concepts into auditable
summaries.
```

### Action surface vs threat action wording

```text
A login form, download button, payment button, wallet connection button,
support contact, or third-party redirect is an action surface, not automatically
a threat action. It becomes a threat action only when combined with deceptive
identity, manipulative narrative, suspicious destination, abnormal submission,
or another high-risk behavior context. Some action surfaces are inherently
high-risk, such as seed phrase/private key collection, remote-control
installation lures, requests to disable security protections, or OTP collection
in an untrusted context.
```
