# TRAINSET_V1.md

> Warden 训练集 V1 规范文档  
> 适用范围：当前已完成的数据结构定义、自动标签脚本、抓取脚本输出以及 V1 阶段训练集生产。  
> 本文档用于冻结 **训练集口径、样本入选规则、切分规则、弱标签使用边界、质检标准**。  
> 本文档不替代 `MODULE_DATA.md`、`MODULE_LABELING.md`、`MODULE_TRAIN.md`，而是作为三者之间的落地桥接文件。

---

## 1. 文档目标

`TRAINSET_V1.md` 的目标不是重新定义整个 Warden，而是明确：

1. **什么样的数据可以进入 V1 训练集**；
2. **进入训练集的样本必须满足哪些字段与质量条件**；
3. **自动规则标签（rule labels）在训练中怎么使用，哪些地方不能乱用**；
4. **训练 / 验证 / 测试集如何切分，避免数据泄漏**；
5. **V1 训练集最终导出成什么样，供后续训练脚本直接消费**。

当前 Warden 已从“狭义钓鱼网站检测”转向 **网页社会工程威胁判断**。因此，训练集 V1 的监督目标也必须同步调整：

- 不是只学“像不像某品牌钓鱼页”；
- 而是学“该网页是否存在社会工程威胁意图、其类型是什么、风险线索来自哪里”。

---

## 2. V1 训练集定位

### 2.1 任务定位

V1 训练集服务于 Warden 的基础监督学习，核心任务为：

- **主任务 A：网页社会工程威胁二分类**  
  判断样本是否属于社会工程威胁网页。

- **辅助任务 B：风险类型 / 意图标签学习**  
  学习网页诱导登录、索要凭证、索要钱包授权、诱导转账、伪客服、伪活动等风险模式。

- **辅助任务 C：品牌 /主题 /页面线索学习**  
  学习页面视觉与文本中出现的品牌仿冒、UI 诱导、表单索取等信号。

### 2.2 非目标

以下内容 **不属于 TRAINSET_V1 的主监督目标**：

- L0 / L1 / L2 级联本身；
- 线上处置策略；
- 多环境对抗采集评估；
- 开放世界完整评测；
- HAR 深度行为图建模；
- 完整 campaign clustering 系统。

说明：

- **L0 / L1 / L2 是推理与处置架构，不是 TRAINSET_V1 的主标签。**
- TRAINSET_V1 重点是把样本与监督信号先做扎实，否则后面所有模型都会学成一锅糊。

---

## 3. 样本定义

### 3.1 基本样本单位

V1 中，**一个样本** 定义为：

> 一次抓取流程得到的、针对某个目标网页落地页的完整采集结果。

一个有效样本通常对应：

- 一个 `sample_id`；
- 一个最终落地 URL；
- 一份页面截图；
- 一份 HTML / DOM 内容；
- 一份元数据；
- 可选的规则标签输出；
- 可选的 HAR / 其他补充材料。

### 3.2 默认模态要求

V1 默认采用以下模态作为训练主输入：

- **截图（必需）**
- **URL / 域名 / 页面基础元数据（必需）**
- **HTML / DOM 文本（必需）**

以下模态在 V1 中为可选：

- HAR
- OCR 中间文件
- 网络请求图
- JS 动态行为结果
- 多环境重复抓取样本

原则很简单：

**先保证“截图 + URL + HTML”三件套稳定，再谈花里胡哨的扩展。**

## 4. 数据来源范围

V1 训练集允许纳入以下来源：

### 4.1 正样本来源

正样本指存在网页社会工程威胁的页面，包括但不限于：

- 仿冒品牌登录页；
- 索要账号密码 / 验证码 / 2FA / 邮箱验证码页面；
- 索要钱包助记词 / 私钥 / 钱包签名授权页面；
- 伪客服、伪工单、伪安全验证页面；
- 诱导转账、充值、解冻、缴费页面；
- 伪空投、伪赠送、伪奖励领取页面；
- 伪下载、伪更新、伪插件安装页面；
- 其他以欺骗、诱导、冒充、强迫交互为核心的网页社会工程样本。

### 4.2 负样本来源

负样本指非社会工程威胁页面，包括但不限于：

- 官方站点真实页面；
- 正常电商 / 社交 / 金融 / Web3 / 工具类网站；
- 普通登录页但无冒充、无异常诱导、无威胁意图；
- 一般营销落地页、公告页、博客页、产品介绍页；
- 与风险品牌相似但确属正常运营的网站。

### 4.3 暂不纳入样本

以下样本先不进入 V1：

- 内容严重损坏、截图空白、HTML 抓取失败的样本；
- 重定向未完成、仅拿到跳转中间页的样本；
- 完全无法判定且关键模态缺失的样本；
- 非网页主体内容，如单纯图片、纯 PDF、无页面上下文的下载链接；
- 与训练目标无关的恶意样本，如纯漏洞利用页、无交互的挂马页等。

### 5.1 与 `MODULE_DATA.md` 的关系

`MODULE_DATA.md` 负责定义数据域、字段、目录和数据生命周期；  
`TRAINSET_V1.md` 负责定义 **哪些数据能被导出为 V1 训练集**。

### 5.2 与 `MODULE_LABELING.md` 的关系

`MODULE_LABELING.md` 负责定义标签体系；  
`TRAINSET_V1.md` 负责定义 **哪些标签是训练主监督、哪些标签只能作为辅助或弱标签**。

### 5.3 与 `MODULE_TRAIN.md` 的关系

`MODULE_TRAIN.md` 负责定义训练模块、输入管线、损失函数和训练策略；  
`TRAINSET_V1.md` 负责定义 **训练前的数据入口规范**。

## 6. V1 标签使用原则

### 6.1 主标签

TRAINSET_V1 至少需要以下主监督字段：

- `label_binary`：是否为社会工程威胁网页；
- `label_source`：该标签来自人工、规则还是混合确认；
- `review_status`：是否已人工复核；
- `split`：train / val / test。

其中：

- `label_binary = 1` 表示存在社会工程威胁；
- `label_binary = 0` 表示正常网页或不构成社会工程威胁。

### 6.2 辅助标签

V1 推荐保留以下辅助标签，用于多任务训练、统计分析或后续蒸馏：

- 风险类型标签（如 login_phish / wallet_phish / fake_support / fake_giveaway / fake_download / payment_fraud 等）；
- 凭证索取标签（是否索要账号、密码、验证码、助记词、私钥、钱包签名等）；
- 品牌标签（brand / brand_confidence / brand_source）；
- 页面语义标签（login / verify / claim / connect_wallet / pay / download / support 等）；
- 表单与交互线索标签；
- 页面语言、站点类别、来源渠道等元数据标签。

### 6.3 L0 / L1 / L2 口径

V1 中明确规定：

- **L0 / L1 / L2 不作为训练集主标签字段的核心监督目标；**
- 它们可以作为后续推理级联、人工复核优先级或样本难度注记；
- 但不能在 TRAINSET_V1 阶段把“模型处置级别”误当成“事实标签”。

否则很容易把“系统设计”与“数据真值”搅成一锅粥，后面全线翻车。

## 7. rule_labels 的使用边界

当前抓取流程已启用 `rule_labels` 输出。V1 对其定义如下：

### 7.1 定位

`rule_labels` 属于：

> **弱标签 / 辅助线索 / 预标注结果**，不是最终人工真值。

### 7.2 允许用途

`rule_labels` 在 V1 中可用于：

- 提高人工标注效率；
- 作为品牌候选提示；
- 作为表单 / 凭证 / 钱包 / 登录等风险线索候选；
- 用于小批量数据的预筛选；
- 用于构造辅助特征或 teacher hint；
- 用于后续分析规则覆盖率与误差来源。

### 7.3 禁止用途

`rule_labels` 在 V1 中 **不得直接等同于人工金标签**，尤其不能：

- 直接拿全部 `rule_labels` 结果当作最终训练真值；
- 直接用纯规则样本构造测试集结论；
- 用未人工复核的 `rule_labels` 样本作为核心评测集；
- 在标签冲突时让规则覆盖人工。

### 7.4 冲突优先级

标签冲突时，优先级固定为：

**人工复核结果 > 明确证据支持的手工修订 > rule_labels > 初始猜测 / 空值**

## 8. 样本入选条件

进入 TRAINSET_V1 的样本，至少应满足以下条件。

### 8.1 必须满足

1. 有稳定唯一标识（如 `sample_id`）；
2. 有最终页面截图；
3. 有最终落地 URL 或可还原的页面地址信息；
4. 有 HTML / DOM 原文或可解析文本；
5. 有基础元数据；
6. 能被映射到冻结字段表；
7. 有可用主标签，至少完成 `label_binary`；
8. 能明确归属到 train / val / test 之一。

### 8.2 建议满足

1. 有 `rule_labels.json`；
2. 有品牌候选与来源标记；
3. 有表单检测结果；
4. 有页面标题、语言、可见文本摘要；
5. 有抓取时间与来源信息。

### 8.3 不可入选情形

以下样本不得进入 V1：

- 截图缺失或损坏；
- HTML 完全空白且无法解释；
- URL 缺失到无法追溯来源；
- 标签缺失且短期无法补齐；
- 与已有样本几乎完全重复且无新增信息；
- 明显是抓取异常页、浏览器报错页、证书错误页、空白中转页。

## 9. 字段冻结原则

V1 执行期间，字段名以既有 **数据结构 V1 文档** 为准，本文件不重新发明字段。

TRAINSET_V1 仅规定：

### 9.1 必备字段组

必须至少能从原始数据中稳定导出以下逻辑字段组：

- **身份字段**：`sample_id` 等；
- **来源字段**：来源目录、采集时间、原始 URL、最终 URL；
- **页面字段**：标题、域名、注册域、页面语言、站点类别；
- **模态字段**：截图路径、HTML 路径、文本路径或可解析文本；
- **标签字段**：`label_binary`、辅助标签、`review_status`、`label_source`；
- **切分字段**：`split`；
- **质量字段**：样本是否完整、是否疑似重复、是否冲突等。

### 9.2 字段变更纪律

在 TRAINSET_V1 生产过程中：

- 不新增临时乱命名字段；
- 不随意重命名已有字段；
- 新增字段先记录到数据结构文档，再进入代码；
- 导出器和训练脚本始终只认冻结字段。

否则后面训练脚本、统计脚本、清洗脚本会互相打架，场面会相当抽象。

## 10. 去重与样本泄漏控制

V1 必须显式控制数据泄漏。

### 10.1 去重粒度

至少进行以下层级去重：

1. **URL 级近重复**：同一页面重复抓取；
2. **截图近重复**：页面几乎相同，仅时间戳、语言或小装饰不同；
3. **HTML 近重复**：DOM 主体高度一致；
4. **域名 / campaign 级重复**：同一诈骗活动批量换壳页面；
5. **品牌模板级重复**：同模板不同子域、不同路径页面。

### 10.2 切分防泄漏规则

原则：

> **同一明显相关页面族，不得同时落入训练集与验证 / 测试集。**

V1 推荐至少按以下优先级控制切分：

- 先按 **注册域（eTLD+1）** 分组；
- 对明显同模板 / 同 campaign 样本再做合并；
- 再进行 train / val / test 切分。

### 10.3 品牌泄漏提醒

如果某品牌仿冒样本过多，且官方站点负样本也很多，必须注意：

- 不能简单随机打散；
- 否则模型会只记住某个品牌表面词，而不是威胁模式。

V1 更推荐：

- 在正负样本中保留品牌多样性；
- 测试集包含未充分暴露的域名 / 页面模板；
- 避免让测试集只是训练集的换皮复读机。

## 11. 训练 / 验证 / 测试切分规范

### 11.1 基本比例

V1 推荐默认切分：

- `train`：70%
- `val`：15%
- `test`：15%

若数据量仍小，可用：

- `train`：80%
- `val`：10%
- `test`：10%

### 11.2 切分原则

切分必须满足：

1. 同一注册域不得跨 split；
2. 同一明显 campaign 不得跨 split；
3. 同一近重复截图 / HTML 不得跨 split；
4. 测试集必须优先保留高质量人工复核样本；
5. 未人工确认的规则样本不得构成测试集主体。

### 11.3 测试集要求

V1 测试集建议具备以下特征：

- 人工复核比例高；
- 品牌分布相对多样；
- 同时包含品牌仿冒类与非品牌型诱导类样本；
- 同时包含正常登录页、正常营销页、正常 Web3 页等负样本；
- 避免测试集全是最简单样本。

---

## 12. 正负样本平衡原则

### 12.1 二分类平衡

V1 不要求严格 1:1，但要求避免极端失衡。

建议：

- 若正样本较少，可适度下采样负样本或训练时加权；
- 若某类负样本过于简单，应提高“相似但正常”的 hard negative 比例；
- 若某品牌正样本过多，需补充更多跨品牌、跨场景样本。

### 12.2 hard negative 定义

V1 应特别重视 hard negatives，例如：

- 正常登录页；
- 正常钱包连接页；
- 正常兑奖 / 活动页；
- 正常下载页；
- 正常客服或工单页；
- 视觉上像“高风险”，但实际正常的网站。

因为没有 hard negative，模型很容易学成“看见登录框就发疯”。这不叫智能，这叫条件反射。

---

## 13. 品牌字段原则

结合当前品牌字典与自动标签情况，V1 对品牌字段作如下约束：

### 13.1 允许空值

`brand` 字段允许为 `none` / 空值，前提是：

- 样本本身并非品牌仿冒型；
- 或当前无法可靠识别品牌；
- 或识别品牌对主标签判断不是必要条件。

### 13.2 品牌不是前置门槛

进入 TRAINSET_V1 的条件是：

> **可判定网页社会工程威胁与否**，而不是“必须识别出品牌”。

这是当前 Warden 路线与传统纯品牌中心方案的重要区别。

### 13.3 品牌字段的推荐用途

- 作为辅助监督；
- 作为统计维度；
- 作为数据清洗与分布分析依据；
- 作为后续 reference list / 轻字典优化的输入。

---

## 14. 样本质量等级

V1 建议给样本增加质量等级，用于训练时筛选和评测集构建。

### 14.1 推荐等级

- **Q1**：截图、HTML、URL 完整，标签人工确认，证据清晰；
- **Q2**：主模态完整，标签可信，少量字段缺失；
- **Q3**：可训练，但依赖规则辅助或部分字段不完整；
- **Q4**：仅保留原始记录，不进入 V1 训练主集。

### 14.2 使用建议

- `train` 可使用 Q1 + Q2 + 部分高可信 Q3；
- `val` 建议以 Q1 / Q2 为主；
- `test` 应尽量使用 Q1，最多少量 Q2。

## 15. 训练集导出格式

TRAINSET_V1 最终应导出为一个稳定、可复现实验的数据快照。

### 15.1 导出目录建议

建议在数据目录下导出：

```text
<data_root>/processed/trainset_v1/
├── manifest_trainset_v1.csv
├── manifest_trainset_v1.parquet
├── splits/
│   ├── train_ids.txt
│   ├── val_ids.txt
│   └── test_ids.txt
├── stats/
│   ├── class_distribution.json
│   ├── brand_distribution.json
│   ├── source_distribution.json
│   └── quality_distribution.json
└── README.md
```

### 15.2 manifest 要求

`manifest_trainset_v1` 至少应包含：

- 样本 ID；
- 各模态路径；
- 主标签；
- 关键辅助标签；
- split；
- label_source；
- review_status；
- quality_grade；
- 去重 / 分组依据；
- 必要元数据。

### 15.3 可复现要求

任何一次导出的 TRAINSET_V1 都必须能回答：

- 来源于哪批原始样本；
- 用了哪版标签表；
- 用了哪版品牌字典；
- 是否启用了 `rule_labels`；
- 去重规则是什么；
- split 是如何生成的。

## 16. 质量检查清单

在导出 TRAINSET_V1 之前，至少完成以下检查：

### 16.1 完整性检查

- 截图路径存在；
- HTML 路径存在；
- URL 字段非空；
- 主标签非空；
- split 非空；
- manifest 可被训练脚本正常读取。

### 16.2 一致性检查

- `label_binary=0` 的样本不应大面积带有明显正样本辅助标签；
- `brand`、`brand_source`、`brand_confidence` 逻辑一致；
- `review_status` 与 `label_source` 不冲突；
- 路径、ID、目录结构与数据文档一致。

### 16.3 泄漏检查

- 同注册域未跨 split；
- 同近重复页面未跨 split；
- 同明显 campaign 未跨 split。

### 16.4 分布检查

- 正负样本比例可接受；
- 风险类型不过于单一；
- 品牌分布不过分塌缩；
- hard negative 占比足够；
- test 集不只是规则最容易命中的样本。

---

## 17. V1 生产纪律

为了避免数据集越做越乱，V1 执行以下纪律：

1. **先冻结字段，再批量生产；**
2. **先做小批导出验证，再扩到全量；**
3. **rule_labels 只做弱标签，不越权；**
4. **优先保证评测集干净可信；**
5. **新增标签需求先记文档，不直接改历史全量；**
6. **不因为某些品牌命中不好，就临时改动训练集主定义。**

品牌字典后续可以增强，离线补标工具也可以继续进化，但 TRAINSET_V1 的核心定义不应因此反复摇摆。

## 18. V1 完成标准（Definition of Done）

当且仅当以下条件满足时，可认为 TRAINSET_V1 完成：

### 18.1 数据层

- 已完成 V1 字段冻结；
- 已形成可复现 manifest；
- 已完成 train / val / test 切分；
- 已完成基本去重与泄漏控制；
- 已产出统计信息。

### 18.2 标签层

- 所有样本至少具备 `label_binary`；
- 测试集高比例为人工复核样本；
- `rule_labels` 已明确标记为弱标签来源；
- 关键辅助标签可用于后续训练实验。

### 18.3 工程层

- 训练脚本可直接读取 TRAINSET_V1 manifest；
- 至少完成一次小规模训练通路验证；
- 训练日志能追溯到 TRAINSET_V1 版本；
- 数据集版本号固定为 `trainset_v1`。

## 19. 当前推荐执行顺序

结合你现在的进度，建议顺序如下：

1. 继续保持当前抓取脚本主逻辑不大改；
2. 用现有字段结构先跑一批小规模样本；
3. 通过离线补标 + 人工检查，把 `label_binary` 先补齐；
4. 验证 `rule_labels` 的覆盖质量，但不拿它直接当金标签；
5. 生成一版小规模 `manifest_trainset_v1`；
6. 做去重、切分、统计；
7. 小跑一次训练链路；
8. 再扩到更大样本规模。

这条路线比较稳，不会把抓取、标注、训练三件事一起拧成麻花。

## 21. 一句话结论

TRAINSET_V1 的核心不是“把已有样本一股脑塞给模型”，而是：

> **在字段冻结、弱标签降级、去重防泄漏、测试集可信这四个前提下，产出一版真正可训练、可复现、可扩展的 Warden 训练集。**
